# REG-DL — Data Lineage Register

| | |
|---|---|
| **Document ID** | REG-DL | **Owner** | Data Governance team (under CDO) |
| **Cadence** | Continuous (auto-discovered + manually maintained) | **Parent framework** | [DGF](../01-frameworks/DGF.md) |
| **Parent standard** | [STD-DG-03 Section 3.3](../03-standards/STD-DG-03-metadata-standard.md) |

## Purpose
Authoritative record of material data flows at GIBB — origin, transformations, consumptions. Primary input to: PDPA data subject requests, regulatory reporting trace, AI training data provenance, incident impact assessment.

## Note on operational form
In practice this register is **tooling-generated** (data catalogue / lineage discovery tools) with human curation for completeness and business-context layer. This document defines the schema and curation expectations.

## Schema
| Field | Description |
|---|---|
| `dl_id` | `DL-NNNN` (or auto-generated by tooling) |
| `data_asset_id` | Link to [REG-DA](REG-DA-data-asset-register.md) |
| `lineage_direction` | Upstream (sources) / Downstream (consumers) |
| `upstream_systems` | Multi reference to CMDB / DA entries |
| `transformations` | Documented transformations applied |
| `downstream_systems` | Multi reference |
| `flow_frequency` | Real-time / Hourly / Daily / Per-event |
| `data_volume_indicator` | |
| `business_glossary_alignment` | Aligned to consistent business definitions |
| `last_discovery_date` | (if auto-discovered) |
| `last_curation_date` | (manual) |
| `completeness_indicator` | Full / Partial / Unconfirmed |

## Worked example
| dl_id | asset | upstream | downstream | flow | completeness |
|---|---|---|---|---|---|
| DL-0001 | DA-0001 Customer master | Branch onboarding system; Online onboarding; CRM | Core banking; Marketing; AML/KYC; Regulatory reporting | Real-time | Full |
| DL-0002 | DA-0002 Account transactions | Core banking; Payments; Branch teller | Customer statements; Analytics; Regulatory reporting; Fraud detection model (AIM-0001 — training input) | Real-time | Full |
| DL-0003 | DA-0007 Fraud model training data | Account transactions + customer master + historical fraud labels | Fraud detection model (AIM-0001) | Per-training cycle | Full |

## Maintenance
- Tooling auto-discovers physical lineage where possible
- Data steward adds business-context layer
- Material change to lineage triggers update + impact assessment
- Lineage gaps flagged per dataset; closure prioritised by criticality
